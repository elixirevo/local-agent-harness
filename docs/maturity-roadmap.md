# 성숙 단계 로드맵 (M1~M4 + 보류 백로그)

> [local-agent-harness-plan.md](local-agent-harness-plan.md)의 Phase 0~4와 백로그 2차분(Phase 5~6)이 완료된 시점(2026-07-07, 38커밋, 208 유닛 테스트)에서, **기능 추가를 멈추고 도구를 성숙시키는 단계**의 계획이다. 원칙: 새 기능은 실사용 근거가 생긴 뒤에만 추가한다.

## 현재 상태 스냅샷

- 완성: 프로바이더 4종(Ollama·llama.cpp·vLLM·MLX, 라이브 검증은 Ollama+MLX), 도구 7종+MCP, 권한 게이트, 루프 가드, 프롬프트 티어, 텍스트 프로토콜 폴백, 컨텍스트 수명 관리(캘리브레이션된 예산·FRC·압축·grammar 재시도), 서브에이전트(Explore/Verify)+read-only 병렬, 계획 모드, 세션 저장/복원, `/remember`, 평가 하네스.
- 검증된 사실: gemma4:e2b temp 0.4가 기준(n=44, 86%), llama3.2(3B)는 텍스트 프로토콜로도 에이전트 작업 천장(3/11), MLX는 cached_tokens를 실제 보고.
- 한계: **실사용 이력 0**. 모든 검증이 우리가 만든 시나리오 안에서 이뤄짐.

---

## M1. 패키징 — 실사용 진입장벽 제거

지금은 저장소 안에서 `npx tsx src/cli/main.ts ...`로만 실행된다. 어디서든 `harness`로 쓰이게 한다.

작업:
1. `npm run build` 산출물 점검 — dist/cli/main.js의 shebang·실행권한, ESM import 경로가 빌드 후에도 유효한지.
2. `npm link` 동작 확인 → 임의 디렉터리에서 `harness -m gemma4:e2b` 성공.
3. `package.json` files 필드 정리(dist만 포함), `private: true` 유지 여부 결정(npm 공개는 별도 결정).
4. 버전 태깅: `git tag v0.1.0`.
5. README에 설치 섹션(빌드 → link) 추가.

**완료 기준**: 하네스 저장소 밖의 아무 프로젝트에서 `harness -p "..."`가 동작. 예상 규모: 소(1시간 미만).

## M2. 도그푸딩 — 실사용으로 마찰 수집

만든 시나리오가 아닌 실제 작업이 진짜 백로그를 만든다. **이 단계의 산출물은 코드가 아니라 마찰 목록이다.**

방법:
1. 대상: 이 하네스 저장소 자체 + 사용자의 다른 실제 프로젝트 1개.
2. 실제 작업을 시킨다 — 테스트 추가, 리팩토링, 문서 수정, 버그 수정. 기준 모델 gemma4:e2b(-M ask로 시작, 신뢰 쌓이면 auto).
3. 마찰이 보일 때마다 즉시 기록: `/remember 마찰: <내용>` (AGENTS.md의 Harness notes에 쌓임). 세션 트랜스크립트(.harness/sessions/)는 재현 근거로 보존.
4. 관찰 항목 체크리스트:
   - 프롬프트: 모델이 지시를 오해하는 지점, 시스템 프롬프트가 부족/과잉인 지점
   - 도구: 에러 메시지가 모델을 복구시키지 못하는 경우, 없어서 아쉬운 도구(WebFetch? MultiEdit?)
   - 권한: ask 피로(불필요하게 자주 묻는 분류), auto가 불안한 지점
   - 컨텍스트: 압축 품질 저하가 체감되는 시점, ctx% 추정과 체감의 괴리
   - UX: 스피너/출력 렌더링, /명령 부족
5. 주기적으로 마찰 목록을 심각도×빈도로 정리 → 다음 작업 선정. 수정은 반드시 `harness eval` 회귀 검증과 함께.

**완료 기준**: 실작업 10회 이상 + 우선순위 매겨진 마찰 목록. 예상 규모: 수일(사용자 주도, 배경 진행).

## M3. 코드 리뷰·정리 패스

6개 phase를 빠르게 쌓아서(38커밋) 전체를 한 번 훑을 가치가 있다. 기능 동결 상태에서 수행.

작업:
1. 버그 헌팅 리뷰: 경계 조건(스트림 파서, 가드 상태, 압축 롤백), 동시성(병렬 실행 도입 후 toolCtx.readFiles 공유), 에러 경로.
2. 단순화 리뷰: 중복 제거, 죽은 코드, 과잉 방어. 특히 **repl.ts가 6 phase에 걸쳐 자란 최대 파일** — 명령 핸들러/렌더링/세션 관리 분리 후보.
3. 테스트 갭 점검: 208개가 있지만 리뷰 중 발견되는 미커버 경로(특히 CLI 계층 — main.ts는 테스트 0).
4. 발견 사항은 correctness 먼저, 정리는 뒤에 — 별도 커밋으로.

**완료 기준**: 리뷰 결과 문서화 + confirmed 이슈 수정 + `eval` 회귀 통과. 예상 규모: 중(반나절).

## M4. 평가 신뢰도 강화

평가가 하네스의 품질 게이트인데, 현재 약점: 모델 2종 편중, 소형 시나리오뿐, 셀당 1런 기본.

작업:
1. 모델 확충: `ollama pull qwen3:8b`(~5GB, 사용자 액션) → 계획서의 원래 기준 라인 확보. 3-모델 매트릭스(gemma4/qwen3/MLX-Qwen3) 기준선 측정·기록.
2. 무거운 시나리오 2~3개: 실제 크기 저장소 스냅샷(예: 이 하네스 소스 복사본)에서 검색+수정, 다중 파일 변경, 긴 테스트 출력 해석.
3. 안정성 관례화: 비교 목적 실행은 `--runs 3` 기본 권장을 README에 명시. 리포트에 셀 간 편차 표기 검토.
4. 기준선 스냅샷: 주요 설정의 성공률을 docs 또는 .harness/eval에 "기준선"으로 남겨 회귀 비교 기준점으로 삼기.

**완료 기준**: 3-모델 × 확장 스위트 기준선 리포트 1부. 예상 규모: 중(구현 반나절 + 컴퓨트).

## 보류 백로그 (Phase 7 — 진입 조건: M2의 실사용 근거)

실사용 마찰이 정당화할 때만 착수한다. 지금 만들면 또 "시나리오용 기능"이 된다.

| 항목 | 내용 | 진입 조건(예시) |
|---|---|---|
| 멀티에이전트 코디네이터 | 배치 병렬을 넘어 워커 위임·조율(문서 §5.3 패턴) | 실사용에서 "컨텍스트 오염 때문에 큰 작업 실패"가 반복 관찰 |
| 세션 메모리 자동 추출 | 종료 시 자동 요약 제안 + y/N 확인(2단계) | /remember를 자주 잊는다는 마찰 기록 |
| MCP sampling/roots | 서버 발신 요청 확장(현재 ping만) | 실사용 MCP 서버가 해당 기능을 요구 |
| llama.cpp 라이브 검증 | `brew install llama.cpp` 후 어댑터·GBNF 실검증 | grammar 강제를 실전에서 쓰려 할 때 |
| WebFetch 등 신규 도구 | 필요 도구 추가 | 실사용에서 부재가 반복 확인 |
| 프롬프트 튜닝 | 티어별 문구 개선 | eval 기준선 대비 개선이 측정으로 입증될 때만 |

## 진행 순서 요약

**M1(1시간) → M2(수일, 사용자 주도) 병행 M3(반나절) → M4(반나절+컴퓨트) → M2 마찰 목록으로 Phase 7 선정.**

M1·M3·M4는 하네스(에이전트)가 단독 수행 가능. M2는 사용자의 실사용이 핵심이고, qwen3:8b pull(M4)만 사용자 액션이 필요하다.
